# 自进化体系

> 隶属：[神圣规划](aimon.md)
> 相关：[世界树](foundation/irminsul.md) · [时执](shades/istaroth.md) · [草神](archons/nahida.md) · [冰神](archons/tsaritsa.md) · [三月](foundation/march.md) · [权限与契约](permissions.md) · [迁移方案](migration.md) · [待办项](todo.md)

**定位**：在现有神圣规划之上，以"**跨会话经验积累 → 行为自我调整**"为目标，沿既有组件加钩子，不新增一轨。

## 一、设计原则

1. **不新增主干**：所有能力落在已定义的组件上（世界树 / 时执 / 派蒙 / 草神 / 冰神 / 三月），只扩能力边界，不改调用拓扑。
2. **进化只发生在会话边界**：严禁 mid-turn 自改 prompt / skill / 记忆。所有写入在时执收尾阶段或三月定时触发时完成。
3. **只存不推**：记忆写入世界树遵循既有"只存不推"原则。消费者（派蒙）按既有"启动读 + 四影通知"两条路径感知；运行时主动查询只发生在**请求入口的 prefetch**。
4. **草神是业务接口，世界树是存储**：记忆的读写业务接口由草神收口，落盘走世界树。与现有知识库的边界完全一致（见 [世界树](foundation/irminsul.md#与草神的边界)）。
5. **写入者单一**：每一类记忆只有一个写入者，避免多写冲突（见下表）。
6. **安全闸门继承**：AI 自举生成的 skill / 新增工具等"进化产物"走既有四影审查通道（见 [新 skill 上线流程](aimon.md#25-新-skill-上线流程)），不开后门。

## 二、能力分层

自进化不是单一机制，而是 **四层叠加**，按落地顺序排列：

| 层级 | 能力 | 核心组件 | 产物 | 优先级 |
|---|---|---|---|---|
| L1 · 经验记忆 | 跨会话记忆积累与召回 | 世界树 + 时执 + 派蒙 + 草神 | `memory/` 数据域 | **MVP** |
| L2 · Prompt 进化 | 七神 persona / 用户偏好化表达自调整 | 草神 + 三月 | 七神 prompt 增量 | Phase 2 |
| L3 · Skill 自举 | AI 生成新 skill | 冰神 + 死执 + 草神 | 新 skill 入世界树 | Phase 3 |
| L4 · 轨迹沉淀 | 为未来 SFT / RL 保留原料 | 时执 + 原石 | 带标签的归档轨迹 | 可选 |

## 三、L1 · 经验记忆（MVP）

### 3.1 记忆四分类

参照业界 agent harness 的通用划分，所有记忆写入前必须归为以下其一，便于定向召回和定向撤销：

| 类型 | 含义 | 典型内容 | 召回时机 |
|---|---|---|---|
| `user` | 用户画像 | 角色 / 长期偏好 / 知识背景 | 每次请求前 prefetch |
| `feedback` | 用户对 AI 行为的显式纠正或确认 | "别再主动修测试"、"就这样保持" | 相关话题出现时 |
| `project` | 项目/话题级非显然事实 | 某仓库的内部约定、跨会话延续的任务状态 | 主题命中时 |
| `reference` | 外部系统指针 | "bug 在 Linear INGEST 项目"、"面板地址" | 主题命中时 |

**条目结构**（落盘字段草案）：

```
id:        uuid
type:      user | feedback | project | reference
subject:   user_id 或 project_scope（用于分片）
title:     短摘要（≤40 字）
body:      正文（≤2KB，feedback/project 需含"为什么"和"何时适用"）
tags:      主题关键词数组（FTS 用）
source:    session_id（溯源）
created:   时间戳
updated:   时间戳
ttl:       可选过期时间（默认永久；feedback 可设 90 天）
```

### 3.2 世界树扩展

在 [世界树](foundation/irminsul.md) 的数据域表里新增一行：

| 数据域 | 可读 | 可写 |
|---|---|---|
| 记忆 `memory/` | 派蒙（启动预热 + 请求前 prefetch）、草神面板（展示 / 编辑）、三月（定时反思读取） | **草神**（唯一写入者，承接时执收尾写入 / 面板人工编辑 / 三月反思合并） |

**目录划分**：

```
irminsul/memory/
  ├── user/{user_id}/
  ├── feedback/{user_id}/
  ├── project/{project_scope}/
  └── reference/{user_id or project_scope}/
```

**索引需求**：每条按 `tags` 建倒排；按 `subject` 分片；预留 FTS（全文检索阈值待定，并入 [todo.md](todo.md) 技术选型）。

### 3.3 时执扩展：收尾提炼

时执在既有 **压缩 → 归档 → 审计** 三步之后，新增一步 **经验提炼**（最靠前，以便被归档覆盖到）：

```text
【时执收尾】
  1. 活跃压缩（已有）
  2. 经验提炼（新增）
      ├─ 将本会话消息链交给草神
      ├─ 草神用浅池 LLM 按四分类抽取候选记忆
      ├─ 草神去重 / 合并 / 决定写入 or 丢弃
      └─ 草神写世界树 memory/*
  3. 最终归档（已有）
  4. 审计复盘（已有）
```

**关键约束**：
- 提炼是**时执触发、草神执行、世界树落盘**，三方职责不越界。
- 用浅池（见 [神之心](foundation/gnosis.md)），控制成本。
- 提炼失败不阻塞归档；失败原因入审计。
- 空会话 / 纯闲聊可跳过提炼（由时执前置判断）。

### 3.4 派蒙扩展：入口 prefetch

[派蒙](paimon/paimon.md) 现有的"轻鉴权 + 轻量安全 + 意图粗分类"之后、路由天使/四影之前，插入：

```text
派蒙 prefetch:
  1. 从本地 memory 缓存（启动时从世界树预热）按 user_id 捞 user 类
  2. 用意图粗分类的主题关键词按 tags 召回 feedback / project / reference
  3. 合并 → 注入 system context（长度预算由时执压缩阈值保障）
```

**缓存策略**：
- 与既有 skill 声明 / 授权记录缓存并列，启动时一次性读入。
- 运行时更新路径 = **四影通知**（时执收尾写入后，由四影广播派蒙刷新对应分片）。
- 与 [权限与契约](permissions.md#画像更新链路) 的"只存不推 + 四影通知"完全一致，不引入订阅。

### 3.5 草神扩展

[草神](archons/nahida.md) 承接三类操作（与既有 `Prompt 调优` / `偏好管理` 职能同根同源）：

| 操作 | 触发方 | 说明 |
|---|---|---|
| 写入记忆 | 时执收尾 | 从会话抽取 → 归一 → 写世界树 |
| 面板编辑 | 用户（通过知识/偏好面板） | 查看 / 改写 / 删除记忆 |
| 反思合并 | 三月定时触发（Phase 2 起） | 聚合近期记忆 → 发现模式 → 更新摘要条目 |

面板能力天然扩展，复用既有授权面板的读写模式。

## 四、L2 · Prompt 进化（Phase 2）

**目标**：基于累计 feedback 微调七神 persona 与对用户的表达风格，不改业务逻辑。

**机制**：

```text
三月（每日定时）→ 草神（反思）
  ├─ 扫 memory/feedback/{uid}/ 近 N 条
  ├─ 聚合出"高频纠正模式"
  ├─ 生成 persona patch（结构化字段：七神 id / 调整维度 / 触发条件）
  └─ 写世界树 memory/persona_patch/{uid}/
```

**消费**：派蒙 prefetch 时把 `persona_patch` 作为 system context 的一部分注入到对应七神的调用前置。七神本体 prompt 不被篡改，只在请求链路上叠加。

**边界**：persona patch **永远不进业务逻辑**（雷神怎么写代码 / 水神评审什么维度不受影响），只影响 **语气 / 选择哪类表达 / 何时主动提醒**。

## 五、L3 · Skill 自举（Phase 3）

对应 [todo.md](todo.md) 第 1 节第 2 项。触发与审查路径**完全复用** [新 skill 上线流程](aimon.md#25-新-skill-上线流程)：

```text
草神（反思中发现）：
  "同一请求模式出现 ≥N 次，且无现成 skill 覆盖"
   └─ 向冰神提交 skill 提案（自然语言 + 用途 + 参数草案）

冰神（AI 自举生成）：
   └─ 生成 skill 代码 + manifest（含 sensitivity 字段）
       → 魔女会 → 死执审查 → 通过 → 冰神写世界树 + 四影通知派蒙

拒绝路径：
   死执拒绝 → 冰神不装载 + 记录 memory/feedback（避免反复提案同一被拒设计）
```

**关键守则**：
- 冰神仍是 skill **唯一写入者**；自举不开后门。
- 自举 skill 的 `sensitivity` 默认按最严（敏感），强制走死执每次审查。
- 提案来源记入 skill manifest（`origin: ai_gen` / `proposed_by_session`），便于回溯和治理。
- 原石给 `purpose=self_evolution` 单独打 tag，防止自举回路失控烧钱（见 [原石](foundation/primogem.md)）。

## 六、L4 · 轨迹沉淀（可选，不入 MVP）

为将来做 SFT / RL 留原料，不做学习本身。

- [时执](shades/istaroth.md) 归档时在条目上打 `trajectory_quality` 标签（由水神评审结果 + 审计结论联合决定）。
- 原石按 `trajectory_id` 聚合 token 消耗。
- 不搭建训练管线，不引入 submodule，不接外部服务。仅保证"未来若要训练，数据是结构化的、可查的、标注的"。

**明确不做**：RL 主循环、微调脚本、rollout 服务。参考对标项目的经验，这些基础设施维护成本极高，当前阶段 ROI 为负。

## 七、权限与安全

| 场景 | 归属 | 说明 |
|---|---|---|
| 记忆写入 | 草神 | 唯一写入者；时执触发 / 面板编辑 / 三月反思三条来源 |
| 记忆读取 | 派蒙（prefetch）、草神（面板 / 反思）、三月（反思） | 派蒙只读本地缓存 |
| 记忆撤销 | 用户（通过草神面板） | 复用授权撤销的 UI 模式 |
| AI 自举 skill | 冰神 | 必过死执，永不跳审（与运行时 plugin 一致） |
| persona patch 注入 | 派蒙 | 仅作用于 system context，不改任何七神本体 prompt |

**敏感度分级**（补充到 [权限与契约 · 敏感度分级](permissions.md#敏感度分级)）：

- 记忆读：普通
- 记忆写（时执自动）：普通（内部触发，不询问用户）
- 记忆写（面板人工）：普通（用户在面板内已授意）
- AI 自举生成 skill：**敏感**（必须死执审查 + 对首次使用询问用户）

按 [migration.md](migration.md) 的 Phase 切分，**本方案不新增 Phase，只往既有 Phase 里塞坑位**：

| Phase | 既有目标 | 本方案动作 |
|---|---|---|
| **0 · 仓库卫生** | — | 本文档归档；在 `todo.md` 引用相关未决项 |

| **B · 七神骨架** | 代理委托老代码 | 冰神 manifest 预留 `origin` 字段；skill `sensitivity` 字段落地（[todo.md](todo.md) 2.7）；草神暴露写记忆接口（空实现） |
| **C · 四影骨架** | 死执→生执→空执→时执 | 时执收尾预留 `extract_learnings` 钩子（占位，可关闭） |

| **E · 神之心分池** | 浅深分层 | 确认草神记忆抽取走浅池、反思合并走浅池、自举 skill 走深池 |

## 九、验收标准（L1 · MVP）

1. 新会话可以用自然语言问"我之前跟你说过我是 X 角色吗" → 派蒙能从 `memory/user/{uid}/` 召回。
2. 用户明确说"以后都用 Y 风格回我" → 时执收尾写入 `feedback`，下次会话派蒙自动遵循，不需用户重复说。
3. 用户在草神面板可查看、编辑、删除任意记忆条目。
4. 任何记忆写入都能在时执审计日志里追溯到源会话（`source: session_id`）。
5. 关闭 `extract_learnings` 钩子（feature flag）时，系统行为退回到无记忆基线，所有既有测试通过。

## 十一、明确不做

- **mid-turn 自改**：不允许 agent 在当前回合内修改自己的 prompt / skill / 记忆。
- **RL / 微调主循环**：当前阶段纯原料化，不建训练基础设施。
- **主动事件推送**：记忆写入不触发事件广播，派蒙不订阅；唯一的运行时更新路径是既有的四影通知。
- **绕过死执**：任何 AI 自举产物必经审查，包括看起来"只是改 prompt"的 persona patch（首次生成须死执过审后方可叠加）。
- **七神本体 prompt 修改**：persona 进化只作用于 system context 叠加层，不篡改七神的 prompt 模板。
