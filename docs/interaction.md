# 交互架构：用户视角的消息流

> 隶属：[神圣规划](aimon.md)
>
> 本文档从**用户视角**描述派蒙与用户的交互时间线：在三条路径（闲聊 / 天使 / 四影）下，用户在 WebUI 和 QQ 分别会看到哪些消息、以什么样的节奏送达。
>
> 实施情况见 [progress.md](progress.md)。

---

## 一、核心思想

### 1.1 两层分离

```
业务层（path 决定打什么标签）
  ┌──────────────────────────────────┐
  │ 闲聊 / 天使 / 四影 的 notice 序列 │   渠道无关
  └──────────────────────────────────┘
                 ↓
渠道层（channel 决定能不能送达）
  ┌──────────────────────────────────┐
  │ Web: 尽量送达 + 弱化样式          │
  │ QQ:  seq 预算内送达 + 窗口外丢弃  │   按能力 degrade
  └──────────────────────────────────┘
```

- **业务代码只打标签**（ack / milestone / tool / thinking / done_recap），不关心 channel 是谁。
- **渠道代码只管送达**：Web 基本全发；QQ 按 5 条 `msg_seq` 预算 + 290s 被动窗口做过滤。
- **同一事件在两渠道的文案完全一致**，差别只在"Web 收到 / QQ 因窗口关而没收到"。

### 1.2 notice 种类速览

| kind | 含义 |
|---|---|
| `ack` | 任务接收即时回执（含任务短标题） |
| `milestone` | 关键阶段切换（死执通过 / 已编排 / 已派发 / 等）|
| `tool` | 单次工具调用（仅 Web） |
| `thinking` | 25s 静默兜底（仅 Web） |
| `done_recap` | 完成 / 失败的最终摘要 |

### 1.3 视觉样式

- **Web**：`notice` 一律渲染为**浅灰小字、无头像、轻缩进**，和正文气泡视觉分层；多条 `thinking` 覆盖同一元素，不堆积。
- **QQ**：每条 notice 都是一条独立 QQ 消息（QQ 没气泡样式可做弱化），所以只发关键事件，不发工具/watchdog 这类琐碎的。

## 二、场景时间线

### 2.1 闲聊（chat）

用户说："你好派蒙"

**Web**：
```
[typing 动画闪烁]
→ 「你」「好」「呀」「，」「旅」「行」「者」...（逐字蹦出）
→ ---
   0.82秒 | ~$0.0003 · 🧠 deepseek-chat
```
若首 token 超 25 秒未出 → 浅灰小字"…还在忙，已工作 25s…"（后续 token 把它覆盖）。

**QQ**：
```
（等 2-3 秒）
↓
你好呀，旅行者！我是派蒙，有什么可以帮你的？
```

闲聊**没有 ack / milestone**。意图分类 < 1s，直接回复正文即可。

---

### 2.2 天使（skill）

用户粘贴："https://www.bilibili.com/video/BV..."

**Web**：
```
┌ [浅灰] 🎯 走 bili，通常 10-30 秒 —— 做：抓取视频信息
├ [浅灰] 🔧 正在调用 web_fetch
├ [正文] 这个视频是...（逐字流式）
└ ---
   12.3秒 | ~$0.002 · 🧠 deepseek-chat
```

若工具静默超 25s → 浅灰"…还在忙，已工作 25s…"。

**QQ**（`tool` / `thinking` 渠道层丢弃）：
```
[seq 1] 🎯 走 bili，通常 10-30 秒 —— 做：抓取视频信息
[seq 2] 这个视频是...（完整一段）
```

天使的"短标题"（上例中"抓取视频信息"）由天使分派时生成（可复用 skill description）。

---

### 2.3 四影（complex）

用户发：`/task 帮我分析一下 xx 股票近期走势并写投资建议`

#### 2.3.1 正常路径（任务 2 分钟跑完）

**Web**（异步后台路径，SSE 在"已进入后台"hint 发完后关闭）：
```
┌ [浅灰] 收到任务：分析 xx 股票走势并写建议，正在准备（安全审查 + 编排）…
├ [浅灰] 🔒 安全审查通过
├ [浅灰] 🚀 已派发 5 个子任务执行：
│          1. 草神 · 收集财报数据
│          2. 草神 · 分析走势
│          3. 水神 · 评审方案
│          4. 草神 · 撰写投资建议
│          5. 时执 · 归档产物
│        如超过几分钟未完成，可随时去 /tasks 面板或 /task-list 查看进度。
├ [正文] ✅ task=abc12345 已授权，进入四影管线后台执行
│         - DAG: 5 节点
│         - 进度/结果推到「📨 推送」会话
│         - 跟踪: /tasks 看状态
└ ---（SSE 关闭，后台继续跑）

之后（异步）：
  - 📨 推送会话收到 🎉 完成 + 完整摘要
  - /tasks 面板状态从"进行中"变为"完成"
```

**关键点**：四影任务是异步后台执行，原会话 SSE 在 prepare 阶段结束时就关闭。
所以**子任务列表必须在 prepare 的最后一条 milestone 里就送达**，
execute 阶段的完成摘要走 📨 推送会话 + /tasks 面板（不走原会话）。

**QQ**（M2 目标状态）：
```
[seq 1] 收到任务：分析 xx 股票走势并写建议，正在准备…
[seq 2] 🔒 安全审查通过
[seq 3] 🚀 已派发 5 个子任务执行：
        1. 草神 · 收集财报数据
        2. 草神 · 分析走势
        3. 水神 · 评审方案
        4. 草神 · 撰写投资建议
        5. 时执 · 归档产物
        如超过几分钟未完成，可随时 /task-list 查看进度。
[seq 4] ✅ task=abc12345 已授权，进入四影管线后台执行（hint）
[seq 5] 🎉 完成｜摘要：xx 股票近期...（仅窗口内触发才发）
```

5 条 `msg_seq` 典型分配。

#### 2.3.2 超窗口（QQ 特有，任务 > 290 秒）

```
[seq 1~4] 同上
（窗口关闭，seq 5 的 done_recap 被丢弃）
```

用户稍后发消息查询：
```
用户: /task-list

派蒙:
📋 最近 7 天任务：
1. [完成] 8 分钟前 · 分析 xx 股票走势并写建议
2. [完成] 昨天 · 帮我重构 auth 模块
3. [失败] 前天 · 爬取 yyy 网站
发送 /task-index 1 查看详情

用户: /task-index 1

派蒙:
📌 任务：分析 xx 股票走势并写建议
状态：完成 · 2026-04-27 19:23 · 耗时 7 分 32 秒
摘要：
xx 股票近期...
```

`/task-index N` 的 N 是 `/task-list` 返回的序号（1 基），TTL 10 分钟。

#### 2.3.3 prepare 阶段失败（死执拒 / 授权全拒）

**Web**：
```
┌ [浅灰] 收到任务：...  ← 这条可能已发（若过了死执）或未发（ack 延迟机制）
└ [正文] ⚠️ 请求未通过安全审查：{原因}
```

**QQ**（因 ack 延迟到过死执才发，prepare 失败时 ack 根本没送出）：
```
[seq 1] ⚠️ 请求未通过安全审查：{原因}
```
只用 1 条 seq。

#### 2.3.4 授权全拒

**Web**：
```
┌ [浅灰] 收到任务：... (ack)
├ [浅灰] 🔒 安全审查通过
├ [独立气泡] 🛡️ 权限询问：task 需要以下 3 个敏感操作...
│           [用户输入"拒绝"]
└ [正文] 任务已取消（你拒绝了全部敏感操作）
```

**QQ**（已支持 ask_user：发询问消息 + 等用户下条入站消息作答）：
```
[seq 1] 收到任务：...
[seq 2] 🔒 安全审查通过
[seq 3] 🛡️ 权限询问：task 需要以下 3 个敏感操作...请回复「同意/放行」继续...
（用户在 290s 窗口内回复"拒绝"）
[seq 4] 任务已取消（你拒绝了全部敏感操作）
```
若用户 30s 不回复 → 保守拒绝 + 取消任务（同步路径）。

#### 2.3.5 执行中失败（七神阶段异常）

**Web**（前 4 条 notice 同正常路径）：
```
...
└ [正文] 💥 失败：{原因}
```

**QQ**（窗口内失败）：
```
[seq 1~4] 同正常
[seq 5] 💥 失败：{原因}
```

**QQ**（窗口外失败）：前 4 条 notice 已发，失败消息丢弃，用户稍后 /task-list 看到"失败"状态。

## 三、渠道能力对照

| 方面 | WebUI | QQ |
|---|---|---|
| 正文样式 | 逐字流式 | 整段一次性 |
| `ack` / `milestone` 样式 | 浅灰小字、无头像、视觉弱化 | 独立消息条（QQ 无样式区分能力） |
| 工具调用提示 `🔧` | 有 | 无 |
| `…还在忙…` watchdog | 有（25s，上限 3 条，同元素覆盖） | 无（seq 宝贵） |
| 四影 `done_recap` 最终摘要 | 一定收到 | 仅 290s 窗口内收到；超出走 /task-list 查 |
| 长任务跟进度 | /tasks 面板 + 📨 推送会话 | /task-list + /task-index |
| 权限询问 `ask_user` | 支持（独立气泡 + 输入框） | 支持（发询问消息 + 等下条入站消息作答；30s 超时保守拒绝） |

## 四、task-list（两渠道共用）

主动查询历史任务的能力。数据同源（世界树 task 表），展现形式按渠道最优。

### QQ

```
用户: /task-list

派蒙:
📋 最近 7 天任务：
1. [完成] 2 小时前 · 分析 xx 股票走势
2. [进行中] 30 分钟前 · 帮我重构 auth 模块
3. [失败] 昨天 · 爬取 yyy 网站
...
发送 /task-index N 查看详情
```

- 最多 20 条，按 `updated_at DESC`
- 筛选：`creator startswith '派蒙'` + `lifecycle_stage != 'archived'` + 7 天内
- 索引仅一次 list 后短暂有效（TTL 10 分钟），重新 list 自动重编号
- 按 channel_key 隔离（不同会话各自的编号缓存互不影响）

### WebUI

`/tasks` 面板双 tab："定时任务"（原有）/「四影任务」：
- 字段：短标题 / 状态 / 创建时间 / 耗时 / 子任务计数（完成/总数/失败）
- 数据源同 `/task-list`（最近 7 天，20 条上限）
- 点击单卡片 → modal 显示 edict + subtasks 表格（assignee / 描述 / verdict / 结果摘录） + 摘要

### 摘要 fallback 链（QQ / WebUI 共用）

`/task-index` 与 WebUI 详情都走 [`paimon/shades/_task_summary.py`](../paimon/shades/_task_summary.py) 同款逻辑，
按"信息密度"逐级回退：

1. workspace `summary.md`（写代码任务专用，时执归档时生成）
2. `push_archive` 里 `actor='四影'` 且 `extra.task_id` 匹配的最终消息（`_compose_final` 的产物，
   含跨节点合成；管线终点 `_notify_progress` 写入）
3. 拼接所有 `completed` subtask.result（逐节点兜底）
4. 诊断兜底：区分"全部 archon 都返回空"（疑似 `_extract_result` 抓不到 LLM 终局消息）vs 普通空状态

## 五、常见误解澄清

- **"QQ 上为什么不做 watchdog？"**——5 条 msg_seq 预算宝贵，每条 "…还在忙…" 都是吃正文名额，不值当。QQ 用户本来就习惯 bot 有 10-30s 等待。
- **"四影 milestone 要不要加耗时估算？"**——`ack` 里说过一次（"预计 3-8 分钟"），后续不重复；估算本身不准，说多了反噬信任。
- **"QQ 窗口外任务完成怎么通知？"**——不主动通知，用户下次发消息时 `/task-list` 自己能看到"完成"状态。主动推送需要腾讯开放平台能力（目前不具备）。
- **"ack 为什么不能立刻发？"**——四影 prepare 阶段就失败（死执拒 / 授权全拒）的话，ack 和 "失败说明" 两条消息会连发，一条就够用。QQ 上每省一条 seq 都是赚的，所以 ack 延迟到过死执再真实送出。
