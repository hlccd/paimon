# character: 跨游戏角色调研

复用 topic skill 的 bili + xhs 中文 UGC 数据，综合"角色机制 / 配队 / 强度 / 抽取建议"。

支持任何有中文社区攻略覆盖的游戏（米哈游、鹰角、库洛、网易、库洛、Fate 系、悠星等）。

## ⛔ 唯一执行路径（违规即重写整个 reply）

**收到 query 后，第一个动作必须是**：调 `exec` 跑

```bash
python3 skills/topic/scripts/research.py "<query>" --sources bili,xhs --days 30 --emit md
```

注意：
- query 原样保留（不要补/删游戏名/角色名；user 输什么就是什么）
- 必须传 `--sources bili,xhs`（character 只用这两源；zhihu/tieba/weibo 攻略含量低，且需 cookies）
- 必须传 `--emit md`（topic 已自己压缩成简报；exec 工具上限 8000 字符，用 json 会被截掉 50%+ 数据）

**不存在以下决策路径**——下面任何一种推理出现都违规，必须重写：

- ❌ 「我不认识这个角色 / 这个游戏」→ 跑脚本（脚本真采集，数据说话）
- ❌ 「这是新角色还没正式上线」→ 跑脚本（爆料 / 前瞻 / PV 解析在 UGC 里就有）
- ❌ 「这个游戏太冷门可能没数据」→ 跑脚本（采集为空就走「数据缺失说明」段照实说）
- ❌ 「先 web_fetch 查一下角色背景」→ 跑脚本（topic 是唯一数据源）
- ❌ 「user 应该用 web-search / topic / chat 而不是 character」→ user 主动选了 character，不替 user 改路径

**严禁的逃逸动作**：跳过 exec 直接基于训练知识答 / 用 web_fetch 自己搜 / 调 use_skill 切其他 skill。

跑完 exec 拿到 stdout 之后，才有资格按下面「输出契约」格式写 reply。

## ⚠️ 输出契约（必须严格遵守，违规即重写）

### 独立调研原则（跨轮隔离）

**每次 character 调用是独立的**——只看本次 user 消息里的 query，**完全忽略 chat history 里之前的调研**。

- ❌ 不允许跨轮对比（连续两次调用时不要做 X vs Y 对比）
- ❌ 不允许引用上一次调研的结论
- ✅ 每次都从零开始：跑 `topic.research.py "<query>"` → 按下面格式写本次结果

### 标准输出结构（结构固定，内容动态）

```markdown
# {query}

## 角色定位
{1 句话总结 + 1-3 个标签}（从 UGC top 内容共识里提炼；如 UGC 信号不一致，写"UGC 评价分歧"）

## 核心机制 / 数值
- {机制点 1}（来自 bili top X 视频共识）
- {机制点 2}
- {机制点 3}
（只列 UGC 共识里反复出现的关键点；**不要**用 LLM 训练知识里的内容补全——版本可能已变）

## 配队 / 阵容建议
1. **{推荐配队 1 名称或定位}**：{角色 A} + {角色 B} + ...（{适用场景}）
2. **{推荐配队 2}**：...
（提取 UGC top 视频里出现频率高的搭配；如 UGC 没明确配队推荐，写"UGC 内容偏个人攻略向，未见明确配队共识"）

## 抽取 / 养成价值
- 强度评估：{基于 UGC 共识，整体倾向}
- 池子价值：{首发 / 复刻 / 双 up / 限定与否——如能从 UGC 抓到}
- ⚠ 通用版不结合个人抽卡进度（如需"结合我祈愿进度"建议，直接说，会走水神特化路径）

## 信源
- bili: {N} 条
- xhs: {N} 条
{若有缺源 / 0 条}: ⚠ {原因，原样照搬 stdout 状态}
```

### ⚠️ 执行流程

调脚本 → 用 stdout 写 reply，两步结束。

1. 调 `exec` 跑 `python3 skills/topic/scripts/research.py "<query>" --sources bili,xhs --days 30 --emit md`
2. stdout 是 markdown：含「## 综合 Top N」（标题 + 链接 + 源标 + 摘要）+「## 各源讨论重点」+「## 各源采集情况」+「各源原始素材」段；character 综合需要的标题 / body / 信源全在
3. 按上面"标准输出结构"重组 reply

### 严禁

- ❌ 顶部加 emoji / 角色扮演开场白（`✨ 调研结果` / `旅行者，派蒙这就帮你查` 等）
- ❌ 用 LLM 训练知识替代 UGC 数据（即使你"很确定"也不行——版本/环境/配队可能已变）
- ❌ 主观夸大评价词（"封神" / "真神" / "逆天" / "版本之子" / "T0 神位" / "毕业级"）
- ❌ 抽取建议里写"建议你抽 X 命 Y 精"——通用版不知道用户进度，**绝不诱导消费**
- ❌ 跨轮引用上次调研内容
- ❌ 跑完脚本再调 web_fetch 补充——stdout 是唯一数据源
- ❌ 把 engagement 数字塞 reply（播放 1.2M 等都不要；只在「信源」段说"N 条"）
- ❌ 改写 UGC 标题（翻译 / 缩短 / 拼接）；引用时直接 `[标题原文](url)`

### 数据缺失降级

如果 stdout 「各源采集情况」段显示所有源 0 条 / 全部 missing_cookies / 采集失败：
- 不要编内容
- reply 直接写：

```
# {query}

⚠ 本次未拉到 UGC 数据。

可能原因：
- 角色名 / 游戏名拼写差异（试试别名）
- 该角色 / 话题在中文社区讨论度低
- bili / xhs 暂时性接口异常

stdout 详情：{原样列「各源采集情况」段}
```

### 输出前 Self-Check（必过）

发送 reply 前逐条核对：

1. [ ] 顶部第一行是 `# {query}`，没 emoji / 角色扮演开场白
2. [ ] 5 段顺序固定：角色定位 → 核心机制 → 配队 → 抽取价值 → 信源
3. [ ] 所有论断都能在 stdout「综合 Top N」/「各源原始素材」里找到对应（**没编 LLM 训练知识**）
4. [ ] 抽取建议里**没**"建议抽 X 命"或类似诱导消费的表达
5. [ ] **没**用主观评价词（封神 / 真神 / 逆天 / T0 / 版本之子 / 毕业级）
6. [ ] 信源段列实际条数；缺源标 ⚠ 并照搬 stdout 错误信息
7. [ ] 全文无 emoji，无角色扮演口吻
8. [ ] 配队推荐（如有）来自 UGC top 视频反复出现的搭配，不自己编
9. [ ] 引用 UGC 标题时是原文（没翻译 / 没缩短 / 没拼接）
