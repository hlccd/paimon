# N+M+K 多轮审查方法论

## 核心命题

**单轮 LLM 审查既不够也不准**：
- **不够**（假阴性）：注意力在长上下文中衰减，一次扫描必然遗漏
- **不准**（假阳性）：LLM 会编造或过度泛化

本方法论通过「**多轮递进发现 + 多轮交叉校验 + [可选] 多轮回归验证**」对抗这两类错误。

---

## 双层循环结构

所有 module 统一使用**外层迭代 × 内层轮次**模型：

```
外层迭代（重复直到连续 clean 或达最大迭代）
│
├── 迭代 1 ──► 迭代 2 ──► ... ──► 迭代 N
│     │
│     ▼
│   ┌──────────┐   ┌──────────┐   ┌──────────┐
│   │  发现     │──►│  验证     │──►│  修复     │
│   │ (n 轮)   │   │ (m 轮)   │   │ 确认问题  │
│   └──────────┘   └──────────┘   └──────────┘
│
└── 停止条件：连续 C 次迭代确认 0 问题 且 已完成 ≥ K 次迭代
```

### 问题发现引擎（21 个问题型 module）

```
发现(n 轮) ──► 验证(m 轮) ──► [fix 模式: 修复 ──► 回归(r 轮)]
```

### 机会发现引擎（extensibility）

```
Scan ──► Analyze ──► Recommend ──► Validate(v 轮) ──► 过滤
```

---

## 发现阶段

尽可能广撒网。此阶段**允许假阳性**，由验证阶段过滤。

### 视角清单

每次迭代的发现阶段必须从以下 **全部 5 种视角** 审查，不可只用其中几种：

| # | 视角 | 侧重点 |
|---|------|--------|
| 1 | 数据流 | 输入如何流经系统，哪里变形、丢失、污染 |
| 2 | 错误路径 | 异常分支、失败回退、资源清理 |
| 3 | 并发 | 共享状态、race condition、顺序依赖 |
| 4 | 契约 | 函数/模块边界的输入输出约定 |
| 5 | 运维 | 可观测性、升级路径、故障恢复 |

多次迭代的目的不是换视角，而是用同样的全视角重新扫描，抓上轮遗漏。

### 分组策略

- 按目录/模块/语义关联分组，每组 10-30 文件
- 相关文件同组（跨文件交互需要上下文）
- 核心模块优先扫

### 输出格式

```
CANDIDATE | {严重度预估} | {check-id} | {file}:{line} | {描述} | {证据}
```

---

## 验证阶段

对候选问题做**严格核验**，过滤假阳性。

### 为什么 M ≥ N

假阳性比假阴性更昂贵——假阳性进入修复阶段可能改坏正确代码。

### 共识机制

- 默认阈值：**≥ 2 轮独立确认 = 真问题**（同一问题在 ≥2 轮不同视角中被标记）
- 仅 1 轮确认 → DEFERRED（标注「仅单轮确认，待人工判断」），不进入修复
- 全部 REJECTED → 移除

### 输出格式

```
CONFIRMED | {最终级别} | {check-id} | {file}:{line} | {描述} | {修复建议}
REJECTED  | {check-id} | {file}:{line} | {驳回理由}
DEFERRED  | {check-id} | {file}:{line} | {需要什么上下文}
```

---

## 修复阶段（仅 fix 模式）

仅在 docs/code-vs-docs 输入模式且 `--fix != report-only` 时执行。code 输入模式和 extensibility module 始终 report-only。

1. **串行修复**（避免文件冲突）
2. **按级别过滤**（`--fix=p0` 只修 P0）
3. **修复后独立复查**：切换视角重新读修改后文件确认正确性
4. 复查失败 → 回滚 + FIX-FAILED
5. **回滚方式**：使用 `Edit` 工具将文件内容恢复至修复前版本（修复前备份到内存）；跨文件修复按修改的逆序逐文件回滚
6. 回滚失败 → 标记 FIX-ROLLBACK-FAILED，记录到 state.json errors 字段，不中止流程

---

## 回归阶段（仅 fix 模式）

对抗 LLM 改坏文档的风险。仅在 fix 模式下执行。

- K=1：基础验证
- K=2：修复周围扩展扫描
- K=3：跨文件调用者检查

发现新问题 → 重新走 V+F+R（**最多 3 次迭代**）。

---

## 停止模式

所有引擎统一使用迭代-至-clean 模式：

每完成一次迭代（发现 + 验证 + 修复），检查本次是否确认了新问题：

- **有新问题**：连续 clean 计数归零，继续下一次迭代
- **无新问题（clean）**：连续 clean 计数 +1

满足以下**两个条件同时成立**时停止：
1. 已完成的迭代次数 ≥ 最少迭代（`min_iter`）
2. 连续 clean 次数 ≥ 要求的连续 clean 次数（`clean_iter`）

若迭代次数达到最大上限（`max_iter`）仍未满足条件，也停止并在报告中标注未收敛。

完整档位对照表见 SKILL.md「深度对照表」。**原则**：验证轮 ≥ 发现轮。

## 机会发现引擎的差异（extensibility module）

extensibility module 不用 N+M+K，改用 **Scan → Analyze → Recommend → Validate** 四阶段。

前三阶段产出带 score 排序的拓展建议。第四阶段做多轮交叉校验：

- **验证轮数**：quick=1, standard=2, deep=3
- **共识机制**：与问题发现引擎相同——≥2 轮独立确认 = 保留推荐
- **严重度分配**：P0=核心缺失、P1=重要缺口、P2=排期实施、P3=锦上添花
- 被 REJECTED 的推荐从最终报告中移除
- 被 DEFERRED 的推荐标注为「需更多上下文」

详见 `modules/extensibility.md` 和 `execution-guide.md § 机会发现引擎`。

## 假阳性抑制规则

验证阶段必须检查候选是否命中以下抑制条件。命中则 REJECTED（标注抑制原因）。

| # | 抑制条件 | 理由 |
|---|---------|------|
| 1 | **纯风格偏好** | 缩进/空行/括号位置/引号风格——属于 linter 职责，非审查范围 |
| 2 | **项目既有模式** | 项目中已广泛使用的模式，即使非"最佳实践"——除非有安全/可靠性风险 |
| 3 | **无证据的微优化** | 无明确性能数据或瓶颈证据的优化建议——除非在已确认的热路径上 |
| 4 | **测试代码重复** | 测试中的重复代码——测试可读性优先于 DRY 原则 |
| 5 | **三方库惯用法** | 第三方库/框架的标准使用方式——除非存在已知安全漏洞 |

> 安全原则：抑制规则不适用于安全类 finding（SEC-*）。安全问题宁可过报不可漏报。

## 反模式

1. **单轮跑完就下结论** —— 噪声太高
2. **验证轮只是重复发现轮** —— 验证的任务是核验不是再次发现
3. **docs fix 模式跳过回归直接结束** —— 改坏风险未知
4. **并行执行修复** —— 文件锁冲突
5. **用发现阶段结论直接生成报告** —— 假阳性会让用户对报告失去信任
6. **extend track 不做 Validate 就输出报告** —— 未经验证的推荐可信度低
