# Module: alignment（方案对齐）

## 核心问题

做没做对？——代码/变更是否忠实实现了方案设计。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code-vs-spec | ALN-001 ~ ALN-014 |
| change-vs-spec | ALN-001 ~ ALN-014（限变更范围内） |
| spec | 不适用（由 feasibility 模块覆盖） |
| code | 不适用 |
| docs | 不适用 |

**特殊性**：本模块在发现阶段增加"方案要点提取"前置步骤（见下方）。

---

## 检查项

### 实现对齐（ALN-001~008）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| ALN-001 | 接口定义对齐 | 方案定义的每个接口在代码中是否完整实现 | P0 |
| ALN-002 | 数据模型对齐 | 字段名/类型/约束是否与方案一致 | P0 |
| ALN-003 | 架构落位对齐 | 代码分层/模块归属是否符合方案设计 | P1 |
| ALN-004 | 外部依赖对齐 | 方案要求的外部调用是否都已实现（含超时/降级/打点） | P1 |
| ALN-005 | 配置项对齐 | 方案定义的配置项是否在代码/配置文件中体现 | P1 |
| ALN-006 | 流程一致性 | 代码执行流程是否与方案流程图/序列图一致 | P0 |
| ALN-007 | 测试覆盖对齐 | 方案要求的测试场景是否有对应测试用例 | P1 |
| ALN-008 | 错误处理对齐 | 异常/降级/兜底逻辑是否与方案设计一致 | P1 |

### 差异检测（ALN-009~012）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| ALN-009 | 缺失实现检测 | 方案明确要求但代码中完全缺失的项 | P0 |
| ALN-010 | 超出方案检测 | 代码中存在但方案未提及的实现 | P2 |
| ALN-011 | 偏差分析 | 实现与方案有差异且无注释/commit 说明原因 | P1 |
| ALN-012 | 方案模糊区域 | 方案描述不清导致实现无据可依的区域 | P2 |
| ALN-013 | 方案缺陷反馈 | 代码实现优于方案设计时标注，建议反向更新方案 | P2 |
| ALN-014 | 规范冲突检测 | 方案设计与项目现有规范/最佳实践冲突时标注 | P1 |

---

## 严重度调整

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| ALN-003（架构落位） | P1 | 核心模块 → **P0** / 辅助模块 → P2 | 核心模块落错层影响全局 |
| ALN-010（超出方案） | P2 | 涉及安全/数据 → **P1** / 纯优化 → P3 | 未经方案审批的安全相关变更需关注 |
| ALN-011（偏差） | P1 | 有代码注释说明 → **P2** / 无解释 → P1 | 有解释的偏差风险降低 |

## 方案冲突处理规则

alignment 审查中会遇到方案本身存在问题的情况，按以下规则处理：

### 1. 方案不完整或模糊时

- 明确标出方案中的模糊点或缺失项（ALN-012）
- 基于项目现有代码和最佳实践给出审查意见
- 在报告中标注"方案待补充"部分，建议完善方案后重新对齐

### 2. 代码实现优于方案设计时

- 在报告中说明代码的改进点及其合理性（ALN-013）
- 建议反向更新方案文档以反映实际实现
- 不因"代码超出方案"而标记为问题

### 3. 方案与项目规范冲突时

- 优先遵循项目现有规范和最佳实践
- 标注方案中的不合规设计（ALN-014）
- 给出符合项目规范的修正建议

## 质量门禁判定

alignment 模块在报告中除 8 维对齐评分外，增加总体判定：

| 判定 | 条件 |
|------|------|
| **APPROVE** | 0 个 P0，对齐总分 ≥ 80 |
| **REQUEST_CHANGES** | 0 个 P0 但存在 P1，或对齐总分 60-79 |
| **BLOCKED** | 存在任何 P0，或对齐总分 < 60 |

## 方案要点提取（前置步骤）

alignment 模块在标准发现阶段前增加此步骤，产出 **spec-points.jsonl**。

### 提取指令

```
【方案要点提取指令】

## 任务
全面阅读方案文档，提取所有可验证的设计要点。

## 提取维度
1. 接口定义：函数/API/端点名称、入参、出参、错误码
2. 数据模型：类/表/结构体名称、字段列表、类型、约束
3. 架构决策：模块划分、分层归属、职责边界
4. 外部依赖：调用的服务/API、超时/降级要求
5. 配置项：配置键名、默认值、环境差异
6. 业务流程：正常流程步骤、分支条件、异常分支
7. 测试要求：测试场景、覆盖率目标、验收标准
8. 错误处理：异常类型、降级策略、兜底行为

## 输出格式（严格 pipe-delimited）
SPEC-POINT | {dimension} | {id} | {description} | {verifiable_criteria} | {source_location}
SUMMARY | 提取 {N} 个设计要点 | 维度分布: {breakdown}
```

### 后续使用

发现阶段逐 SPEC-POINT 比对代码，CANDIDATE 附带 spec-point ID 以追溯。

---

## 发现指令

### 视角轮换

1. **逐点比对**：从 spec-points.jsonl 出发，逐点在代码中验证实现
2. **代码遍历**：从代码结构出发，检查每个实现是否有方案依据（检测超出方案的部分）
3. **流程追踪**：从方案流程图出发，在代码中追踪完整执行路径
4. **边界检查**：聚焦方案中的异常/边界/降级设计，验证代码覆盖
5. **配置核验**：方案配置项 vs 代码配置文件逐项比对

### 专有关注点

1. **零容忍缺失**：方案明确要求的任何内容在代码中缺失即 ALN-009（P0）
2. **偏差需解释**：实现与方案不一致时，代码注释或 commit message 中应有说明
3. **额外实现需评估**：代码超出方案的部分，评估必要性和风险
4. **方案模糊标注**：方案本身描述不清的区域要明确标出，建议方案补充

### 变更模式（change-vs-spec）特殊处理

- 只验证变更涉及的方案要点，非全量对齐
- git diff 文件自动限定审查范围
- 新增文件：验证是否属于方案要求的新模块
- 修改文件：验证修改方向是否与方案一致
- 删除文件：验证删除是否符合方案要求的重构/清理

---

## 验证指令

- 必须同时读方案和代码——不能只凭一方判断
- 接口对齐需要精确到参数级别——方法名匹配不等于接口对齐
- 流程对齐需要追踪完整路径——不能只看入口和出口
- 超出方案的检测要区分"合理补充"和"擅自变更"——补充错误处理通常合理，新增未经审批的功能不合理
- 方案模糊区域要给出具体建议——不能只说"方案不清"，要说"这里需要定义 X"

## 报告侧重

### 对齐评分（8 维度，满分 100）

| 维度 | 满分 | 评分标准 |
|------|------|---------|
| 接口定义 | 10 | 每个方案接口在代码中完整实现 |
| 数据模型 | 10 | 字段/类型/约束完全一致 |
| 架构落位 | 15 | 分层/归属/职责边界正确 |
| 外部依赖 | 15 | 调用/超时/降级/打点齐全 |
| 配置项 | 10 | 键名/默认值/环境隔离正确 |
| 流程一致 | 15 | 正常+异常流程完全匹配 |
| 测试覆盖 | 15 | 方案场景在测试中有对应 |
| 错误处理 | 10 | 降级/兜底/重试与方案一致 |

### 报告分段

1. **缺失实现**（方案要求但未实现）——按严重度排序
2. **超出方案**（代码存在但方案未提及）——按风险评估排序
3. **偏差分析**（实现与方案不一致）——附方案原文 vs 实际实现对比
4. **方案待补充**（方案模糊区域）——附具体化建议
