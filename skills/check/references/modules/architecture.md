# Module: architecture（架构）

## 核心问题

结构好不好？——分层是否清晰、耦合是否合理、能否可控地演进。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | ARCH-001 ~ ARCH-012, DEAD-006, DEAD-008, COR-008 |
| docs | 不适用 |
| code-vs-docs | ARCH-001 ~ ARCH-012, DEAD-006, DEAD-008, COR-008（仅审查代码侧） |

---

## 检查项

### 代码输入

**主检查（ARCH，12 项）**：ARCH-001 ~ ARCH-012 全部

**引用检查**：
- DEAD-006（重复实现）— 架构冗余
- DEAD-008（未使用的模块）— 架构遗留
- COR-008（字符串字面量 vs Enum）— 反映抽象薄弱
- PRJ-014（循环依赖检测）— import 环是架构缺陷

### 文档输入

不适用。架构是代码结构特性，纯文档审查无法评估。

### 交叉输入（code-vs-docs）

仅审查代码侧。激活全部代码检查项。

---

## 严重度调整

架构问题以 P1-P2 为主，除非严重阻塞开发：

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| ARCH-001（循环依赖） | P1 | **P0** | 会阻塞 import 或引发隐藏 bug |
| ARCH-005（紧耦合） | P2 | >1000 行模块 → **P1** | |

---

## 发现指令

### 视角轮换

1. **数据流**：追踪数据在层间流动——有没有跨层直接访问？数据结构是否在各层之间变形？
2. **错误路径**：异常在层间如何传播——是被统一处理还是层层 catch 再 raise？
3. **并发**：全局状态——有几个模块在读写同一份状态？能否找到单一所有者？
4. **契约**：每个模块的公开接口——职责清晰吗？能一句话描述做什么吗？
5. **运维**：替换成本——如果要换掉存储/换掉 LLM provider/换掉 channel，要改多少文件？

### 专有关注点

1. 循环依赖：A import B，B 直接或间接 import A——常导致动态 import hack
2. 上帝对象：单个类/模块承担过多职责，改一处影响多处
3. 数据结构泄漏：ORM 对象被直接返回给前端；内部 dict 结构被外部依赖
4. 硬编码依赖：难以在测试中替换
5. 重复实现：同样的功能在 N 处独立实现，修 bug 要改 N 次
6. 不一致的错误处理：同一错误在不同层次行为不同

---

## 验证指令

- 架构判断要给依据——不能只说"耦合太深"，要说明具体哪些模块以什么方式耦合
- 要区分「规模造成」和「设计造成」——大项目必然复杂，关键是看是否符合一致的规则
- 要考虑演进路径——即便当前架构有缺陷，如果重构成本 >> 收益，不应标为 P0

---

## 报告侧重

- 报告头部画一个「架构健康度」表格：循环依赖数、上帝对象数、重复实现数等
- 每个 ARCH finding 附「影响半径」：改进后能让多少模块变清晰
- 按「改造成本 vs 收益」排序推荐修复优先级
