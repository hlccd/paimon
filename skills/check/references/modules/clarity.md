# Module: clarity（清晰度）

## 核心问题

能不能看懂？——代码接口是否清晰、文档表述是否简明、解释是否到位。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | CLR-001 ~ CLR-004, ARCH-007, ARCH-002, ARCH-006, ARCH-010, COR-008 |
| docs | BRV-001 ~ BRV-005, LNG-001 ~ LNG-003 |
| code-vs-docs | 全部（代码侧 + 文档侧） |

---

## 检查项

### 代码输入（9 项）

**主检查（CLR，4 项）**：

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| CLR-001 | 嵌套层级过深 | >4 层嵌套 | P2 |
| CLR-002 | 不直观的布尔参数 | 多个 positional bool 参数 | P2 |
| CLR-003 | 超大函数 | >100 行，可读性差 | P2 |
| CLR-004 | 命名误导 | 变量/函数名与实际行为不符 | P1 |

**引用检查（5 项）**：

| 引用 ID | 清晰度视角 |
|---------|-----------|
| ARCH-007 | 接口契约不清晰——无 type hints + 无 docstring |
| ARCH-002 | 上帝对象——职责不可一句话描述 |
| ARCH-006 | 职责泄漏——业务逻辑散落非业务层，读者找不到 |
| ARCH-010 | 配置与代码混合——行为来源不透明 |
| COR-008 | 枚举值用字符串字面量——含义不自明 |

> ARCH-002/006/010 同时归属 architecture module。clarity 关注"看不看得懂"，architecture 关注"结构好不好"。

### 文档输入（8 项）

- BRV-001 ~ BRV-005 全部（简洁凝练）
- LNG-001 ~ LNG-003 全部（语言质量）

**禁止词列表**（文档输入时检查）：
- 可能、应该、推测、大概、也许、或许、似乎
- 看起来、据推测、理论上、按理说
- 笔者认为、个人理解
- 详见代码、参见源码（文档应自包含）
- 肯定能、绝对不会（过度承诺）
- 简单地、只需要（低估复杂度）
- 显而易见、众所周知（跳过论证）

### 交叉输入（code-vs-docs）

代码侧和文档侧全部激活。额外关注：代码注释/docstring 与外部文档的表述是否一致。

## 严重度调整

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| LNG-003（术语混用） | P2 | **P1**（面向终端用户的文档） | 混淆更严重 |
| ARCH-007（接口契约不清晰） | P2 | P2 | 保持 |
| BRV-002（空洞描述） | P1 | P1 | 保持 |

## 发现指令

### 视角轮换（代码输入）

1. **接口视角**：对每个公开函数/类——能否仅凭签名和名称理解用途？需要读实现吗？
2. **命名视角**：变量名/函数名/类名——是否准确描述其内容/行为？有无误导？
3. **职责视角**：每个模块/类——能用一句话描述其职责吗？读者看到文件名能猜到内容吗？

### 视角轮换（文档输入）

1. **首次阅读者视角**：以零背景知识阅读——哪些地方需要猜测？
2. **简洁性视角**：逐段审查——删掉后丢不丢信息？能用表格替代吗？
3. **术语视角**：全文检索同义词——同一概念用了几种称呼？

### 专有关注点

**代码侧**：
1. 无 type hints 的公开接口
2. 超大函数/类（>100 行函数、>500 行类）
3. 嵌套层级过深（>4 层）
4. 不直观的布尔参数（`do_thing(True, False, True)`）

**文档侧**：
1. 冗余重复段落
2. 空洞铺垫/过渡废话
3. 信息密度低的大段文字（应改表格/列表）
4. 推测性用语

---

## 验证指令

**代码侧**：
- 判断要给依据——不能只说"命名不好"，要说明具体哪里误导
- 要区分风格偏好和真正的清晰度问题

**文档侧**：
- 用"删除测试"验证——删掉后读者是否丢失信息？
- 禁止词必须逐个 Grep 确认

---

## 报告侧重

- 代码 finding 附**改善建议**：具体的重命名/拆分方案
- 文档 finding 附**修改前后对比**：原文 vs 建议版本
- 直接告诉用户"怎么改"而非"哪里差"
