# Module: coherence（连贯性）

## 核心问题

各部分连没连对？——多文件之间的引用链是否闭合、声明是否一致、有无孤儿和陈旧。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | COH-001~004, COH-006~008, COH-010 |
| docs | COH-001~002, COH-004~006, COH-009~010 |
| spec | COH-001~002, COH-004~006, COH-009~010 |
| code-vs-docs | COH-001~010 全部 |
| code-vs-spec | COH-001~004, COH-006~008, COH-010（代码侧） |
| change-vs-spec | COH-001~004, COH-006~008, COH-010（变更范围） |
| code-vs-code | COH-001~004, COH-006~008, COH-010 |
| docs-vs-docs | COH-001~002, COH-004~006, COH-009~010（双侧） |

## 检查项

### 跨文件一致性（COH-001, COH-007）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| COH-001 | 跨文件矛盾 | 同一实体/数值/规则在不同文件中描述不一致 | P1 |
| COH-007 | 跨入口一致性 | README/配置/注册表等多入口文件对同一事实的声明是否一致 | P1 |

### 引用链（COH-002, COH-003, COH-009）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| COH-002 | 引用链完整性 | 配置/文档/代码中引用的文件/模块/资源是否全部存在 | P0 |
| COH-003 | 注册一致性 | 注册表/清单/配置文件中的条目与实际文件是否一一对应 | P1 |
| COH-009 | 缺失交叉引用 | 文档提及实体/概念名称但未链接到对应文件 | P2 |

### 孤儿与陈旧（COH-004, COH-005）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| COH-004 | 孤儿检测 | 存在但无任何其他文件链接/引用/import 指向的文件 | P2 |
| COH-005 | 陈旧内容 | 含日期结论超 90 天未更新；TBD/待定未跟进；引用已删除的文件 | P1 |

### 结构与组织（COH-006, COH-008, COH-010）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| COH-006 | 索引/目录缺失 | 文件存在但未在索引或目录结构文件中注册 | P2 |
| COH-008 | 结构错位 | 文件是否按项目约定的目录模式/命名规范放置 | P2 |
| COH-010 | 建议合并/拆分 | 同一概念在 3+ 文件中重复描述但无专属文件；或单文件 >500 行应考虑拆分 | P3 |

**引用检查**：
- CON-001（跨文档概念一致性——coherence 侧重拓扑结构，consistency 侧重语义自洽）
- PRJ-009（CI 配置引用的脚本/路径存在——也是引用链完整性）

## 严重度调整

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| COH-001（跨文件矛盾） | P1 | 涉及数值/版本号 → **P0** / 描述性差异 → P1 | 数值矛盾直接误导 |
| COH-003（注册一致性） | P1 | 注册表有条目但文件不存在 → **P0** / 文件存在但未注册 → P2 | 前者导致运行时失败 |
| COH-004（孤儿） | P2 | 可执行文件/脚本 → **P1** / 文档/资源 → P3 | 孤儿脚本可能是安全风险 |
| COH-005（陈旧） | P1 | 含确定性断言 → **P0** / 含"计划"/"预期" → P2 | 过时的确定性声明比过时的计划更有害 |
| COH-008（错位） | P2 | 影响自动发现/路由 → **P1** / 仅影响规范性 → P3 | 自动发现依赖目录结构 |

## 发现指令

### 视角轮换

1. **引用追踪**：从注册表/配置/索引入口出发，逐条验证引用的文件是否存在、路径是否正确
2. **反向追踪**：从实际文件列表出发，反查每个文件是否被注册/引用/链接（找孤儿）
3. **一致性比对**：提取多个入口文件对同一实体的声明，检查是否一致
4. **时间线索**：搜索日期标记、TBD/待定/暂定/后续补充，检查是否已跟进

### 专有关注点

1. **注册表识别**：package.json/requirements.txt/_registry.json/index 文件/路由配置/Makefile targets 等
2. **引用链递归**：A 引用 B，B 引用 C——需要递归验证整条链
3. **路径类型**：区分相对路径、绝对路径、变量路径（${VAR}/path）
4. **矛盾检测优先级**：数值 > 版本号 > 状态声明 > 描述性文字
5. **陈旧信号词**：TBD、待定、暂定、后续补充、计划中、预计、TODO
6. **孤儿排除**：.gitignore、LICENSE、CHANGELOG、配置模板不算孤儿
7. **目录约定推断**：从项目现有结构推断命名/放置约定，不依赖外部规范

---

## 验证指令

- 矛盾必须给出具体的两处位置和各自的值——不能只说"不一致"
- 引用链需要区分"构建时引用"和"运行时引用"——两者都算
- 路径引用需考虑变量展开——`${SKILL_DIR}/foo.md` 在变量值已知时应展开验证
- 孤儿判定需要全项目搜索——Grep 文件名确认无任何引用
- 陈旧内容需要区分"过时"和"稳定"——一个 2 年没改的架构文档可能是因为架构稳定
- 跨入口一致性需要语义比对——"支持 Python 3.10+" 和 "requires Python >= 3.10" 是一致的

---

## 报告侧重

- 矛盾 finding 附**两处原文对比**：位置 A 说什么 vs 位置 B 说什么
- 引用链 finding 附**断裂位置**：A → B → ✗C，具体哪个环节断了
- 孤儿按类型分组：代码文件 / 配置文件 / 文档 / 脚本 / 其他
- 输出连贯性指标：引用链完整率 / 注册覆盖率 / 孤儿率 / 矛盾数
