# Module: completeness（完整性）

## 核心问题

全不全？——该有的有没有。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | CMP-006, CMP-007 |
| docs | CMP-001 ~ CMP-005 |
| code-vs-docs | CMP-001 ~ CMP-007 全部 |

---

## 检查项

### 文档输入

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| CMP-001 | 核心公开接口/方法未文档化 | 列出公开 API 比对文档覆盖 | P0 |
| CMP-002 | 错误处理场景未文档化 | 用户遇到报错不知怎么办 | P1 |
| CMP-003 | 配置项未文档化 | 代码读取的 config 在文档中无说明 | P1 |
| CMP-004 | 关键用例未覆盖 | 只有 happy path 无 edge case 说明 | P2 |
| CMP-005 | 缺少环境变量说明 | getenv 调用在文档中无对应 | P1 |

### 代码输入

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| CMP-006 | switch/match 缺少 default 分支 | 枚举分支不完整 | P1 |
| CMP-007 | 函数缺少空/null/异常输入处理 | 公开函数无输入验证 | P1 |

### 交叉输入

全部激活。文档侧检查覆盖面，代码侧检查分支完整性。

**引用检查**：
- ACC-011（核心接口有文档覆盖——与 CMP-001 互补，ACC-011 侧重准确性，CMP-001 侧重覆盖率）
- COR-002（边界条件未处理——代码完整性视角）
- USB-004（错误码/异常无对应文档——完整性视角是"该有的文档没有"）

---

## 发现指令

### 视角轮换

1. **接口覆盖**：列出所有公开 API/方法，逐个检查文档中是否有对应描述
2. **错误路径覆盖**：每个可能的错误/异常，文档中是否有说明和解决方案
3. **配置覆盖**：代码中读取的每个配置项/环境变量，文档中是否有说明

---

## 验证指令

- 必须实际 Grep/Glob 确认——不能凭文档内容推断"已覆盖"
- 内部/私有接口不要求文档覆盖——只关注公开 API

---

## 报告侧重

- 列出「缺失清单」：具体哪些接口/配置/场景缺少文档
- 按影响面排序：核心接口 > 配置 > 边缘场景
