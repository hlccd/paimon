# Module: feasibility（可实施性）

## 核心问题

做不做得出来？——方案的技术假设是否成立，设计是否足够完整以支撑实施。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| spec | FEA-001 ~ FEA-016 |
| code | 不适用 |
| docs | 不适用 |
| code-vs-spec | FEA-005/006/007/012/015（方案侧，为 alignment 提供上下文） |
| change-vs-spec | FEA-005/006/007/012/015（方案侧） |

---

## 检查项

### 技术可行性（FEA-001~004）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-001 | 技术假设可行性 | 方案依赖的技术/API/能力是否真实可用 | P0 |
| FEA-002 | 依赖可用性 | 声明的外部依赖/三方服务是否存在、是否维护中 | P1 |
| FEA-003 | 性能目标合理性 | 性能指标（QPS/延迟/吞吐）与架构设计是否匹配 | P1 |
| FEA-004 | 资源需求合理性 | 人力/时间/基础设施假设是否现实 | P1 |

### 设计完整性（FEA-005~009）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-005 | 接口契约完整性 | 所有对外接口是否定义了入参/出参/错误码 | P0 |
| FEA-006 | 数据模型完整性 | 数据结构/Schema 是否完整定义字段、类型、约束 | P0 |
| FEA-007 | 异常流程覆盖 | 错误/降级/超时/重试场景是否有设计 | P1 |
| FEA-008 | 安全设计充分性 | 数据隔离/传输安全/权限校验/敏感数据处理/认证授权是否有明确设计 | P1 |
| FEA-009 | 可测试性设计 | 是否定义测试场景和验收标准 | P1 |

### 上线保障（FEA-010~011）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-010 | 部署方案完整性 | 上线步骤/环境配置/灰度策略是否定义 | P2 |
| FEA-011 | 回滚方案 | 是否有失败回滚路径 | P1 |

### 文档结构（FEA-013~014）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-013 | 文档结构合规 | 方案是否遵循组织约定的模板/章节结构 | P1 |
| FEA-014 | 强制要素齐全 | 关键章节是否齐全：架构设计/接口定义/数据模型/异常处理/测试策略/部署方案 | P0 |

### 质量闭环（FEA-015~016）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-015 | 逻辑闭环性 | 每个功能的正常+异常流程是否都有明确出口，无悬空分支 | P1 |
| FEA-016 | 测试策略覆盖 | 正常/边界/异常/并发场景是否有对应测试策略或验收标准 | P1 |

### 表述质量（FEA-012）

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FEA-012 | 模糊词检测 | 是否存在无法直接转化为代码的模糊描述 | P1 |

**模糊词列表**：

> 智能分析、相应处理、优化体验、适当调整、合理范围、自动适配、按需处理、灵活配置、简单实现、显而易见、高效处理、统一管理、自动识别、合理设计、适当优化、酌情调整、后续优化、视情况而定、尽可能、若有必要、一般来说、通常情况下、基本满足

## 严重度调整

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| FEA-004（资源需求） | P1 | 有量化数据 → P2 / 纯定性 → **P0** | 无量化的资源估算无法评审 |
| FEA-010（部署方案） | P2 | 核心服务 → **P1** / 内部工具 → P2 | 核心服务上线必须有方案 |
| FEA-012（模糊词） | P1 | 影响接口/数据模型 → **P0** / 影响描述性文字 → P2 | 接口定义模糊直接阻塞开发 |

## 发现指令

### 3 阶段递进（每轮发现按阶段顺序执行）

**阶段 1：结构检查**（FEA-013/014）
- 方案骨架是否完整？章节结构是否符合约定？
- 关键要素（架构/接口/数据模型/异常/测试/部署）是否齐全？
- 若骨架严重缺失（FEA-014 P0），后续阶段仍继续但标注"结构不完整，建议先补全骨架"

**阶段 2：设计审查**（FEA-001~012）
- 技术假设是否成立？依赖是否可用？性能目标是否可达？
- 接口/数据模型是否完整定义？异常/安全/部署是否有设计？
- 模糊词检测

**阶段 3：质量审查**（FEA-015~016）
- 每个功能的正常+异常流程是否闭环？
- 测试策略是否覆盖正常/边界/异常/并发场景？

### 视角轮换

1. **开发者视角**：拿到这份方案能否直接开始编码？哪些地方需要回头确认？
2. **架构师视角**：技术选型是否合理？依赖链是否健康？性能设计是否可达？
3. **QA 视角**：方案是否可测试？验收标准是否明确？异常场景是否覆盖？
4. **运维视角**：上线有没有方案？出问题怎么回滚？监控怎么做？
5. **安全视角**：认证/授权/数据保护是否考虑？

### 专有关注点

1. **接口定义精确度**：入参/出参每个字段是否有类型、是否标注必选/可选、是否定义错误码
2. **数据模型完备度**：字段名/类型/约束/索引是否完整，关联关系是否明确
3. **流程闭环性**：正常流程 + 每个异常分支是否都有明确出口，无悬空分支
4. **依赖可行性**：外部 API 是否真实存在、是否有文档、是否有 SLA
5. **模糊表述**：逐段检查"动词+形容词"组合，能否直接转化为 if/else 或 API 调用
6. **测试策略 5 方法覆盖**：等价类划分/边界值分析/场景法/错误猜测/判定表——方案的测试策略是否覆盖这些方法
7. **安全必检 4 项**：数据隔离/传输安全/权限校验/敏感信息处理——方案是否有明确设计

## 验证指令

- 必须区分"未定义"和"刻意留白"——方案可能显式标注"待定"或"后续迭代"
- 模糊词需要上下文判断——"合理"在定量场景是模糊的，在策略描述中可能可接受
- 性能目标需要对照架构——声称 10ms 响应但设计了 3 次串行外部调用，不合理

## 报告侧重

- 按"阻塞开发 / 影响质量 / 建议改进"三档分组
- 每个 finding 附**缺失内容的建议补充模板**（方便方案作者直接填写）
- 模糊词 finding 附**具体化建议**：原文 → 建议改为
