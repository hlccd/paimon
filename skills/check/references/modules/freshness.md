# Module: freshness（新鲜度）

## 核心问题

新不新？——内容/依赖/API 是否过时。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | FRS-002, FRS-003, FRS-006 |
| docs | FRS-001, FRS-004, FRS-005 |
| code-vs-docs | FRS-001 ~ FRS-006 全部 |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| FRS-001 | 文档描述已删除/重命名的功能 | Grep 源码确认功能是否仍存在 | P0 |
| FRS-002 | 依赖库已停止维护或存在已知 CVE | 检查依赖版本和维护状态 | P1 |
| FRS-003 | 代码使用已废弃 API/方法 | Grep deprecated 标记 | P1 |
| FRS-004 | 文档引用的外部资源已下线 | URL 合理性判断 | P1 |
| FRS-005 | 文档最后更新与代码最后修改差距 >6 月 | git log 比对 | P2 |
| FRS-006 | 使用过时的语言特性/模式 | 如 Python 2 兼容代码在 Python 3 项目中 | P2 |

**引用检查**：
- DEAD-007（过时 TODO/FIXME——freshness 与 hygiene 的交集）
- PRJ-005（依赖声明与 import 一致——过时依赖视角）

---

## 发现指令

### 视角轮换

1. **功能存在性**：文档描述的每个功能在代码中是否仍存在
2. **依赖健康度**：检查核心依赖的维护状态和版本
3. **时间线索**：寻找日期标记（注释中的日期、TODO 时间戳）判断陈旧程度

---

## 验证指令

- "过时"需要证据——不能凭"看起来旧"判断
- 区分"刻意保留的兼容代码"和"遗忘的过时代码"

---

## 报告侧重

- 按影响面排序：面向用户的过时内容 > 内部过时代码
- 标注"过时原因"：功能已删/重命名/API 变更/依赖停维
