# Module: maintainability（可维护性）

## 核心问题

好不好改？——修改代码的成本和风险有多高。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | MNT-001 ~ MNT-004, MNT-006 |
| docs | MNT-005（文档与代码同步成本） |
| code-vs-docs | MNT-001 ~ MNT-006 全部 |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| MNT-001 | 变更放大 | 修改一个业务规则需同步改 >3 个文件 | P2 |
| MNT-002 | 缺少变更隔离 | 无 feature flag/策略模式导致每次变更全量发布 | P2 |
| MNT-003 | 魔法值散落 | 配置/阈值/常量散布代码各处而非集中管理 | P2 |
| MNT-004 | 缺少迁移工具/脚本 | schema 变更需要手动操作 | P2 |
| MNT-005 | 文档与代码同步成本高 | 文档中硬编码代码细节导致每次改代码都要改文档 | P2 |
| MNT-006 | 测试脆弱性 | 测试依赖具体实现细节而非行为/接口 | P2 |

**引用检查**：
- DEAD-006（重复实现——改一处要同步改 N 处）
- ARCH-001（循环依赖——改动牵连不可控）
- ARCH-002（上帝对象——改动影响面大）
- ARCH-005（紧耦合——改动传播）
- ARCH-011（错误处理层级混乱——改错误逻辑要跨层）
- ARCH-012（事件驱动失控——改动追踪困难）

---

## 与 architecture / clarity 的分界线

- **architecture**：结构对不对（是非判断）
- **clarity**：看不看得懂（读的成本）
- **maintainability**：改不改得动（改的成本）——好架构是高可维护性的充分非必要条件

---

## 发现指令

### 视角轮换

1. **变更影响**：如果修改某个业务规则，需要改几个文件？影响几个模块？
2. **回归风险**：改动后有没有自动化手段确认没有引入新问题？
3. **常量管理**：阈值/配置/常量是集中管理还是散落各处？

---

## 验证指令

- 变更放大必须量化——具体说明改 X 需要同步改 Y1/Y2/Y3
- 区分"有意的重复"（如多环境配置）和"无意的散落"

---

## 报告侧重

- 每个 finding 附**变更场景**：在什么情况下这个问题会造成痛苦
- 修复建议按 ROI 排序：低成本高收益的集中化/提取优先
