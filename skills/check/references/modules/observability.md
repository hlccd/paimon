# Module: observability（可观测性）

## 核心问题

能不能观测？——出了问题能不能快速定位。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | OBS-001 ~ OBS-007 |
| docs | 不适用 |
| code-vs-docs | OBS-001 ~ OBS-007（代码侧） |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| OBS-001 | 关键业务操作缺少日志 | 成功/失败均无记录 | P1 |
| OBS-002 | 错误日志缺少上下文 | 只有消息没有请求 ID/用户/参数 | P1 |
| OBS-003 | 缺少健康检查端点 | 无 /health 或 /readiness | P2 |
| OBS-004 | 缺少请求追踪 | 无 trace-id/correlation-id 传播 | P2 |
| OBS-005 | 日志结构不统一 | 部分 JSON 部分纯文本混杂 | P2 |
| OBS-006 | 缺少关键指标暴露 | QPS/延迟/错误率无采集点 | P2 |
| OBS-007 | 异步任务无状态可查 | fire-and-forget 后无法知道结果 | P1 |

**引用检查**：
- REL-001（异常吞没无日志——可观测性视角是"该有日志的地方没日志"）
- REL-017（日志脱敏缺失——"日志有但打了不该打的"）
- PERF-010（日志级别过低——"噪声淹没信号"）
- CST-004（日志格式不一致——"结构化程度不统一影响日志分析"）

---

## 发现指令

### 视角轮换

1. **日志视角**：关键操作路径（创建/删除/修改/登录）有无日志？日志包含什么？
2. **诊断视角**：生产出问题时，凭现有日志/指标能否定位到具体请求和原因？
3. **监控视角**：有无健康检查、指标暴露、告警集成点？

---

## 验证指令

- 必须读实际代码确认日志存在/不存在
- 区分"不需要日志的内部辅助函数"和"必须有日志的业务关键路径"

---

## 报告侧重

- 按关键路径列出日志覆盖情况
- 每个 finding 标注影响：「此处无日志意味着生产出 X 问题时无法定位」
