# Module: performance（性能）

## 核心问题

快不快？省不省？——找出影响响应时间、吞吐量、内存占用的代码。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | PERF-001 ~ PERF-010 + 引用检查 |
| docs | 不适用 |
| code-vs-docs | PERF-001 ~ PERF-010 + 引用检查（仅审查代码侧） |

---

## 检查项

### 代码输入

**主检查（PERF，10 项）**：PERF-001 ~ PERF-010 全部

**引用检查**：
- REL-005（资源泄漏）— 也是内存性能问题
- REL-012（超时未设置）— 影响吞吐
- ARCH-009（同步/异步混用）— 阻塞性能杀手

### 文档输入

不适用。性能是运行时特性，纯文档审查无法评估。

### 交叉输入（code-vs-docs）

仅审查代码侧。激活全部代码检查项。

## 严重度调整

性能问题级别依赖「热路径 vs 冷路径」：

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| PERF-001（N+1） | P1 | 热路径 **P0** / 冷路径 P2 | |
| PERF-002（阻塞异步） | P1 | **P0** | 阻塞整个 event loop |
| PERF-009（正则未预编译） | P3 | 热路径 **P2** / 冷路径 P3 | |

## 发现指令

### 视角轮换

1. **数据流**：追踪数据规模——每个循环/查询处理的数据量是否有上限？
2. **错误路径**：重试/退避逻辑——紧密重试会放大负载
3. **并发**：是否有 connection pool？数据库连接是否会被耗尽？
4. **契约**：公开 API 的 SLA——有文档声明的响应时间吗？代码能保证吗？
5. **运维**：内存增长——哪些结构长期保留？是否有 LRU/TTL？

### 专有关注点

1. N+1 查询：循环里查 DB，特别注意 ORM 的懒加载陷阱
2. 阻塞异步：`time.sleep`、`requests.get`、`open().read()` 在 async 函数中
3. 缓存无上限：`_cache = {}` 没 LRU、没 TTL，长期运行必然 OOM
4. 大对象在热路径：f-string 格式化 dict、log.debug 打印整个 response
5. 正则预编译：热路径用 `re.search(pattern, text)` 而非预编译 pattern
6. 同步锁在异步代码：threading.Lock 用在 asyncio 代码——应该用 asyncio.Lock

---

## 验证指令

- 必须估算规模——每个 PERF finding 附 n 的估计量级（100？10k？1M？）
- 必须识别热路径——区分是每请求执行还是每日一次
- 性能判断要给数据——宁可粗估，不要只说"慢"
