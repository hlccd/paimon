# Module: portability（可移植性）

## 核心问题

能不能迁？——换个环境/平台能不能跑。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | PRT-001 ~ PRT-008 |
| docs | 不适用 |
| code-vs-docs | PRT-001 ~ PRT-008（代码侧） |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| PRT-001 | 硬编码绝对路径 | Grep /home/、/usr/、C:\ 等 | P1 |
| PRT-002 | 硬编码主机名/端口/URL | localhost:8080 等写死在代码中 | P1 |
| PRT-003 | 平台特定系统调用 | os.fork/signal.SIGKILL 等在 Windows 不可用 | P2 |
| PRT-004 | 数据库方言绑定 | 使用特定 DB 特有语法无抽象层 | P2 |
| PRT-005 | 字符编码假设 | 假设 UTF-8 但未显式声明 | P3 |
| PRT-006 | 文件系统大小写假设 | macOS 不区分但 Linux 区分 | P2 |
| PRT-007 | 缺少容器化配置 | 无 Dockerfile/docker-compose 且项目有多依赖服务 | P3 |
| PRT-008 | 用户可见字符串硬编码 | 无 i18n 机制，用户界面文本写死在代码中 | P2 |

**引用检查**：
- ARCH-010（配置与代码混合——硬编码是可移植性障碍）
- PRJ-013（Dockerfile 引用——容器化正确性）
- PRJ-008（.env.example 与 getenv 一致——环境变量是可移植性基础）

---

## 发现指令

### 视角轮换

1. **路径/URL 视角**：Grep 硬编码的绝对路径、localhost、具体域名/IP
2. **平台视角**：识别平台特定 API 调用、文件系统假设
3. **环境视角**：配置是否通过环境变量/配置文件注入，还是写死在代码中

---

## 验证指令

- 区分"故意限定平台"（如明确只支持 Linux 的系统工具）和"无意的平台假设"
- 开发用的硬编码（如测试中的 localhost）不算——只关注生产代码

---

## 报告侧重

- 按环境类型分组：路径 / 网络 / 平台 / 数据库 / 编码
- 修复建议指出应用什么替代（环境变量、Path 库、抽象层）
