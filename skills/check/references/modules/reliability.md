# Module: reliability（可靠性）

## 核心问题

会不会崩/丢/卡？——找出会导致崩溃、数据丢失、静默损坏、资源耗尽的问题。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | REL-001 ~ REL-020 + 引用检查 |
| docs | 不适用 |
| code-vs-docs | REL-001 ~ REL-020 + 引用检查（仅审查代码侧） |

---

## 检查项

### 代码输入

**主检查（REL，20 项）**：REL-001 ~ REL-020 全部

**引用检查**：
- COR-001（状态机非法转换）— 状态混乱常导致数据损坏
- COR-007（空值处理错误）— None 传播是崩溃常见原因
- COR-014（日期时间 None 处理）
- SEC-001（SQL 注入）— 也会导致数据损坏
- ARCH-004（全局可变状态）— 并发风险源头
- ARCH-009（同步/异步混用）— 可能造成死锁

### 文档输入

不适用。可靠性是运行时特性，纯文档审查无法评估。

### 交叉输入（code-vs-docs）

仅审查代码侧。激活全部代码检查项。

---

## 严重度调整

本 module 强调**宁可过报，不可漏报**：

| 检查项 | 默认 | 本 module | 理由 |
|--------|------|----------|------|
| REL-006（悬空 task） | P1 | **P0** | 异步任务悬空在生产会积累 |
| REL-013（重试逻辑缺陷） | P2 | **P1** | 不当重试会放大故障 |

## 发现指令

### 视角轮换

1. **数据流**：追踪每个数据写入点——数据是如何从内存到磁盘/DB 的？中间有任何可能丢失或损坏的环节吗？
2. **错误路径**：所有 try/except/finally——异常发生时，资源是否清理？状态是否一致？部分完成的操作如何回滚？
3. **并发**：所有共享状态（模块级变量、实例属性、数据库行）——有几个 writer？是否存在 check-then-act 的竞态？
4. **契约**：所有跨模块调用——参数/返回值/异常的契约是什么？调用方的假设会被违反吗？
5. **运维**：故障恢复——进程重启后状态如何？未完成的操作会如何处理？是否有孤儿数据？

### 专有关注点（「静默损坏」模式）

1. INSERT OR REPLACE / ON CONFLICT：有 FK CASCADE 的表上会导致关联数据被静默删除
2. 非原子文件写入：直接 write_text 而非 tmp+rename，崩溃时文件残缺
3. ID 生成碰撞概率：生日悖论下，N² 个 ID 在 2^(bits/2) 个样本后必然冲突
4. 异常吞没：`except Exception:` + 无日志 + 继续执行关键逻辑
5. 未 await 的协程：fire-and-forget 场景下可能静默失败
6. 资源泄漏的累积：单次不严重，长期积累导致进程崩溃

---

## 验证指令

- 必须读实际代码——不能只凭发现阶段的描述判断
- 必须考虑时序——尤其是异步/多线程场景，画出可能的 interleaving
- 必须考虑边界——并发为 0、1、N 三种场景都要考虑
- 宁可升级不要降级——如果不确定 P0 还是 P1，选 P0

## 报告侧重

- P0 问题一个也不能漏
- 每个 P0 附**触发场景描述**（什么条件下会真实发生）
- 修复建议具体到代码改动，不能只是"应该加 try/except"
