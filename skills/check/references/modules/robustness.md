# Module: robustness（健壮性）

## 核心问题

扛不扛得住？——遇到意外输入和极端条件时能否优雅降级。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | ROB-001 ~ ROB-007 |
| docs | 不适用 |
| code-vs-docs | ROB-001 ~ ROB-007（代码侧） |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| ROB-001 | 缺少输入验证/sanitization | 用户输入直接进入业务逻辑 | P1 |
| ROB-002 | 缺少熔断/断路器机制 | 外部依赖持续失败时无保护 | P2 |
| ROB-003 | 缺少速率限制 | 无限请求可打垮系统 | P1 |
| ROB-004 | 异常恢复后状态不一致 | 降级后恢复时状态未正确重置 | P1 |
| ROB-005 | 大批量输入无分页/限制 | 一次传入大量数据直接 OOM | P1 |
| ROB-006 | 级联失败无隔离 | 一个下游故障拖垮所有请求 | P1 |
| ROB-007 | 缺少优雅关闭 | SIGTERM 后正在处理的请求直接丢弃 | P1 |

**引用检查**：
- REL-018（依赖服务不可用无降级）
- REL-013（重试逻辑缺陷）
- REL-012（超时未设置）
- COR-002（边界条件未处理）
- COR-004（类型强制转换错误——意外输入）
- REL-016（反序列化错误未处理——意外数据格式）

---

## 与 reliability 的分界线

- **reliability**：正常路径和已知异常路径下系统稳不稳（底线）
- **robustness**：意外打击（恶意输入、外部故障、极端负载）下系统扛不扛得住（韧性）

---

## 发现指令

### 视角轮换

1. **恶意输入**：如果用户故意传入极端值/超大数据/非法格式，会怎样？
2. **外部故障**：如果某个下游服务持续超时/返回错误，系统行为如何？
3. **极端负载**：如果并发量突增 10 倍，哪里会先崩？

---

## 验证指令

- 必须区分"理论上可能"和"实际会发生"——按实际暴露面评估级别
- 内部工具/脚本的 robustness 要求低于面向用户的 API

---

## 报告侧重

- 每个 finding 附**攻击/故障场景**：什么条件下会触发
- 修复建议要具体：加什么限制/加在哪里
