# Module: security（安全性）

## 核心问题

能不能被攻击？——找出可被攻击者利用的漏洞。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | SEC-001 ~ SEC-017 + 引用检查 |
| docs | SEC-005, SEC-006（文档中的代码示例可能泄露密钥） |
| code-vs-docs | SEC-001 ~ SEC-017 + 引用检查 |

---

## 检查项

### 代码输入

**主检查（SEC，17 项）**：SEC-001 ~ SEC-017 全部

**引用检查**：
- REL-016（未处理的反序列化错误）— 可能放大安全问题
- REL-017（日志脱敏缺失）
- COR-004（类型强制转换）— 输入校验薄弱
- PRJ-015（git 中无敏感文件）— 密钥/凭证泄漏到版本控制

### 文档输入

仅检查文档中的代码示例是否泄露敏感信息：
- SEC-005（硬编码密钥出现在文档代码示例中）
- SEC-006（密钥出现在文档的日志示例中）

### 交叉输入（code-vs-docs）

代码侧全部检查。文档侧检查代码示例中的敏感信息。

## 发现指令

### 视角轮换

1. **输入侧**：识别所有 untrusted input 入口——HTTP 请求、CLI 参数、文件上传、第三方 API 响应——追踪它们如何被使用
2. **错误路径**：错误响应是否泄漏内部信息？traceback、SQL、文件路径是否回传给前端
3. **认证**：race condition 导致的越权——session 切换、token 刷新、多设备登录
4. **授权**：每个受保护资源的访问路径——有没有绕过授权的途径？IDOR？缺失的 admin 校验？
5. **密钥管理**：密钥/token/证书——硬编码？配置文件是否入库？日志中是否出现？

### 专有关注点

1. 用户输入拼字符串：SQL、命令、路径、URL、正则——任何字符串拼接都要警惕
2. 密钥管理：源码、配置、日志、错误信息——四处检查
3. 反序列化：pickle、yaml.load、eval——不信任的输入直接反序列化是灾难
4. 授权校验：横向越权（IDOR）比纵向（admin check）更难发现
5. 依赖库漏洞：发现 CVE 已知库应当标记

---

## 验证指令

- 必须给出攻击场景——每个确认的安全问题描述：谁能利用？怎么利用？影响范围？
- 分级要考虑「已暴露 vs 潜在」——漏洞是否已在生产暴露、git 历史是否已入库
- 密钥类问题要立即处理——即使修复，也要建议撤销/轮换

- P0 安全问题**必须置顶**于报告
