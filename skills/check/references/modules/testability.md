# Module: testability（可测试性）

## 核心问题

好不好测？——代码能否方便地编写和维护测试。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | TST-001 ~ TST-007 |
| docs | 不适用 |
| code-vs-docs | TST-001 ~ TST-007（代码侧） |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| TST-001 | 关键路径缺少测试覆盖 | Grep 测试文件中对核心函数的调用 | P1 |
| TST-002 | 函数内硬编码外部依赖无注入点 | HTTP/DB/文件系统直接调用无法 mock | P2 |
| TST-003 | 测试间共享可变状态 | fixture/全局变量导致测试互相污染 | P1 |
| TST-004 | 业务逻辑与 I/O 混杂 | 无法纯单元测试业务逻辑 | P2 |
| TST-005 | 缺少测试工具/fixture 支持 | 无 conftest/factory/mock helper | P2 |
| TST-006 | 异步代码测试困难 | 缺少异步测试基础设施 | P2 |
| TST-007 | 关键分支仅靠集成测试覆盖 | 应有单元测试保证的逻辑依赖 E2E | P2 |

**引用检查**：
- PRJ-011（测试文件非空壳）
- ARCH-005（紧耦合——难 mock）
- ARCH-004（全局可变状态——测试间污染）
- ARCH-010（配置与代码混合——测试时无法控制环境）

---

## 发现指令

### 视角轮换

1. **覆盖率视角**：核心业务函数有没有对应测试？关键分支有没有被测试命中？
2. **隔离性视角**：测试能否独立运行？有无共享状态导致顺序依赖？
3. **mock 友好度视角**：外部依赖能否被替换？有无依赖注入点？

---

## 验证指令

- 必须 Grep 测试目录确认测试存在/不存在
- "难测"需要具体说明——指出具体哪个依赖无法替换、哪个状态会污染

---

## 报告侧重

- 列出「关键未测路径」清单
- 每个 finding 附**改善建议**：如何引入注入点/拆分逻辑
