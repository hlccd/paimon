# Module: usability（易用性）

## 核心问题

好不好用？——用户使用这个项目/API/文档的摩擦有多大。

## 适用输入模式

| 输入模式 | 激活检查项 |
|---------|-----------|
| code | USB-001, USB-002, USB-005, USB-007, USB-008 |
| docs | USB-003, USB-004, USB-006 |
| code-vs-docs | USB-001 ~ USB-008 全部 |

---

## 检查项

| ID | 检查项 | 验证方法 | 默认级别 |
|----|--------|---------|---------|
| USB-001 | API 错误信息不具可操作性 | 检查异常/错误返回是否说明原因和解决方案 | P1 |
| USB-002 | 配置过于复杂 | 必填项过多/默认值不合理/无 example | P2 |
| USB-003 | 缺少快速开始示例 | 用户无法在 5 分钟内跑通 | P1 |
| USB-004 | 错误码/异常无对应文档 | 用户遇到报错无处可查 | P1 |
| USB-005 | CLI 帮助信息不完整或与实际不符 | --help 输出 vs 实际行为 | P1 |
| USB-006 | 文档结构不便查找 | 深层嵌套/无索引/无搜索提示 | P2 |
| USB-007 | 破坏性操作无确认/无撤销 | 删除/覆盖/重置操作无二次确认 | P0 |
| USB-008 | 无障碍访问缺失 | 交互元素缺少 a11y 属性/标签/键盘导航支持 | P2 |

---

## 发现指令

### 视角轮换

1. **新用户视角**：以零背景知识尝试安装、配置、运行——在哪里卡住？
2. **错误场景视角**：故意触发错误——报错信息能否指导修复？
3. **日常使用视角**：常见操作路径——步骤是否过多？有无快捷方式？

---

## 验证指令

- 必须从用户视角判断——不是"代码上能不能做到"，而是"用户能不能搞定"
- 区分"不便"和"不能"——P0 留给"不能"，P1/P2 留给"不便"

---

## 报告侧重

- 每个 finding 附**用户场景**：在什么情况下用户会遇到这个问题
- 修复建议要面向用户体验，不只是代码改动
