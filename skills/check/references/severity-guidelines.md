# 严重度分级标准（Severity Guidelines）

所有 Agent 输出的 finding 必须使用以下四级分级。核心判定口径：**站在「真实用户/运维人员/文档读者」角度，这个问题发生时的代价有多大？**

本标准统一适用于所有输入模式和 module。extensibility module 的 severity 含义见本文件末尾的「extensibility module 严重度映射」。

---

## P0 — 阻断级（必须立即修复）

**判定原则**：用户按文档/正常流程使用时会**直接失败**，或导致**不可恢复的损失**。

### 代码类（Track=code）

- **崩溃/卡死**：进程退出、死锁、无限循环
- **数据丢失**：持久化非原子、覆盖写入、误删除
- **数据损坏**：并发条件下状态不一致、事务边界错误
- **安全泄漏**：密钥硬编码（已入库）、SQL 注入、越权访问
- **契约违反**：API 返回值类型/结构与声明不符导致调用方失败
- **ID 碰撞**：主键使用易冲突方式（如 `random.randint` 小范围）
- **资源泄漏（严重）**：大对象持续泄漏导致 OOM
- **异常吞没（关键路径）**：try/except: pass 包住关键业务逻辑

### 文档类（Track=docs）

- **方法名/函数名拼写错误**：读者在代码中找不到
- **参数名或类型错误**：按文档调用会直接报错
- **文件路径不存在**：读者无法定位代码
- **行为描述与代码逻辑完全相反**
- **内部链接断裂**（404）
- **虚构的类名、模块名、API 端点**
- **已删除功能仍在文档中描述**
- **核心公开接口/方法完全未记录**

### 项目健康度（project-health，含 Skill 合规）

- **SKILL.md 缺失必填 frontmatter 字段**（name/description）
- **name 字段与目录名不一致**（skill 无法被正确激活）
- **README 声明的入口文件/命令不存在**（用户按文档操作失败）
- **配置示例键名与代码读取的键名不一致**（配置不生效）
- **依赖声明缺少代码中实际 import 的包**（安装后无法运行）

### 判定问句

> 「用户遇到这个问题时，能否通过重试/调整输入/查文档自行恢复？」
> - 不能 → **P0**
> - 可以，但要技术介入 → 考虑 P1

---

## P1 — 重要级（应优先修复）

**判定原则**：不会立即失败，但**显著影响可靠性/维护性/理解准确性**，长期累积会演变成事故或严重误解。

### 代码类

- **异常处理不充分**：捕获后只打 log，但后续逻辑依赖该步骤成功
- **状态不一致（可恢复）**：重启能修复，运行中状态混乱
- **资源泄漏（轻度）**：缓慢增长，需定期重启
- **性能严重下降**：N+1 查询、O(n²) 热路径、阻塞主线程
- **边界条件未处理**：空列表、超长字符串、零值分母
- **并发风险（低概率）**：小概率竞态，但触发后果严重
- **可观测性缺失（关键路径）**：关键故障无日志

### 文档类

- **默认值记录错误**：功能可用但行为与预期不同
- **跨文档数值不一致**：如一处说"7 张表"另一处说"6 张表"
- **现状描述与代码不符**（技术方案中）
- **遗漏核心受影响模块**（方案影响面严重低估）
- **整段空洞描述，删除后不丢失信息**
- **意义不清晰的描述，读者无法理解具体含义**

### 项目健康度类

- **allowed-tools 声明了未使用的工具 / 使用了未声明的工具**
- **check-id 在 catalog 中不存在但被 profile 引用**
- **README 安装命令引用的依赖在 requirements 中不存在**
- **版本号在多处声明但值不一致**
- **CI 配置引用的脚本/路径不存在**

### 判定问句

> 「这个问题是否会让排查/运维/理解成本显著增加？」
> - 是 → **P1**
> - 否，只是质量问题 → 考虑 P2

---

## P2 — 改进级（排期修复）

**判定原则**：属于**质量问题**，短期无明显影响，但不改会持续累积技术债或降低效率。

### 代码类

- **架构问题**：模块职责混乱、循环依赖、上帝对象
- **命名/抽象不清**：变量/函数名误导、接口语义模糊
- **重复代码**：DRY 违反、逻辑散落多处
- **缺少测试覆盖**：关键路径无测试
- **弱类型声明**：大量 `any` / `dict`

### 文档类

- **推测性用语**（"可能"、"应该"）代替确定性描述
- **配置项名称或层级描述不准确**
- **非核心方法签名细节错误**
- **架构描述与代码有偏差但不影响核心理解**
- **同一信息在文档内大段重复表述**
- **性能数据无来源标注**

### 项目健康度类

- **catalog 中存在死 ID**（无任何 profile 引用）
- **template 占位符未在填充逻辑中覆盖**（报告会出现原始 `{xxx}`）
- **profile 结构不完备**（缺少某个标准段落）
- **依赖声明中存在代码未 import 的包**（多余依赖）
- **测试文件为空壳**（存在但无实质内容）

### 判定问句

> 「这个问题短期（3 个月内）是否会导致事故或阻塞开发？」
> - 否 → **P2**
> - 是 → 升级到 P1

---

## P3 — 建议级（按需修复）

**判定原则**：只影响**阅读体验或风格一致性**，功能/准确性完全正确。

### 代码类

- **格式问题**：缩进不一致、尾随空格
- **拼写/语法**：变量名、注释中的错别字
- **冗余注释**：无意义的 `# 加一` 之类
- **风格不一致**：单双引号混用

### 文档类

- **格式/排版问题**：表格对齐、缺少空行
- **不精确但无害**：如"约 600 行"实际 613 行
- **缺失非关键细节**：可选参数未列出但不影响核心用法
- **个别冗余过渡句**
- **轻微啰嗦但不影响理解**

### 项目健康度类

- **SKILL.md 行数略超 500 但不严重**
- **LICENSE 文件存在但与 pyproject.toml 声明有大小写差异**
- **死代码：单个未使用变量或 import**

### 判定问句

> 「不改，对未来开发者有实际影响吗？」
> - 没有 → **P3**
> - 有轻微影响 → 考虑 P2

## 边界案例判定表

### 代码类边界

| 场景 | 分级 | 理由 |
|------|------|------|
| 持久化非原子写入，当前无并发写入者 | P0 | 一旦出现并发或崩溃即数据丢失 |
| 小概率竞态（万分之一触发） | P1 | 概率低但生产环境迟早遇到 |
| 异常吞没但有日志 | P1 | 有可观测性救济 |
| 异常吞没无日志 | P0 | 错误完全不可见 |
| `random.randint(1, 999)` 作为 ID | P0 | 小范围随机必然碰撞 |
| `INSERT OR REPLACE` 触发 CASCADE DELETE | P0 | 静默数据丢失 |
| O(n²) 但 n 永远 < 100 | P3 | 无实际性能影响 |
| O(n²) 且 n 可达 10000 | P1 | 大数据下严重变慢 |
| 硬编码密钥在源码（已入库） | P0 | 已扩散到 git 历史 |
| 硬编码密钥在配置文件（未入库） | P1 | 风险存在但未扩散 |
| 测试代码中的 bug | 原级别 -1 档 | 除非 CI 基础设施 |
| 死代码：整个未使用模块 | P2 | 误导后来者 |
| 死代码：单个未使用变量 | P3 | 清理即可 |

### 文档类边界

| 场景 | 分级 | 理由 |
|------|------|------|
| 方法存在但参数顺序记错 | P0 | 位置参数顺序错误导致调用失败 |
| 可选参数默认值记错 | P1 | 功能可用但行为与预期不同 |
| 缺少一个不常用的可选参数 | P3 | 核心用法不受影响 |
| 两篇文档对同一常量给出不同数值 | P1 | 至少有一处错误 |
| 一处数值精确一处用"约" | P3 | 模糊但不矛盾 |
| 外部 URL 完全虚构 | P0 | 误导读者 |
| 文档描述了已删除的功能 | P0 | 读者尝试不存在的功能 |
| 文档未描述新增的核心接口 | P0 | 核心接口缺失 |
| 文档未描述新增的非核心功能 | P3 | 核心功能覆盖即可 |
| 整段内容删除后不丢失信息 | P1 | 占据篇幅不提供价值 |

### 项目健康度类边界

| 场景 | 分级 | 理由 |
|------|------|------|
| name 字段含大写字母 | P0 | 违反 agentskills.io 规范，skill 无法注册 |
| name 含连续连字符 `--` | P0 | 同上 |
| allowed-tools 用了 YAML 数组格式 | P1 | 不符规范但部分 runtime 可能容忍 |
| description 超 1024 字符 | P1 | 违反规范 |
| profile 缺少 Severity Overrides 段 | P2 | 结构不完备但不影响运行 |
| README 声称"支持 Python 3.10+"但实际需 3.12+ | P0 | 用户按文档安装后运行失败 |
| CI 配置中 step name 拼写错误但路径正确 | P3 | 不影响执行 |
| .env.example 多了一个代码未读取的变量 | P3 | 多余但无害 |

## feasibility module 严重度映射

feasibility module 的 finding 关注方案的可实施性缺陷。P0-P3 判定基于「缺失是否阻塞开发」：

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| **P0** | **阻塞开发** | 开发者拿到方案无法开始编码——接口未定义、数据模型缺失、技术假设不成立 |
| **P1** | **影响质量** | 能开始编码但必然产出低质量实现——异常流程未设计、安全未考虑、模糊描述 |
| **P2** | **建议补充** | 方案可执行但存在改进空间——部署方案缺失、回滚未考虑 |
| **P3** | **锦上添花** | 对方案质量有帮助但不影响实施 |

### 边界案例

| 场景 | 分级 | 理由 |
|------|------|------|
| 接口只有名称无参数定义 | P0 | 无法编码 |
| 接口有参数但缺少错误码定义 | P1 | 能编码但异常处理无据可依 |
| 方案说"调用 XX 服务"但未说明超时/降级 | P1 | 可编码但稳定性无保障 |
| 方案说"合理的超时时间" | P1（FEA-012 模糊词） | 无法确定具体值 |
| 方案标注"待定，后续迭代补充" | P3 | 显式留白，非遗漏 |
| 性能目标写 10ms 但架构设计含 3 次串行外部调用 | P0 | 技术假设不成立 |

---

## alignment module 严重度映射

alignment module 的 finding 关注实现与方案的偏差。P0-P3 判定基于「偏差是否导致功能不符预期」：

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| **P0** | **方案要求未实现** | 方案明确要求的功能/接口/流程在代码中完全缺失 |
| **P1** | **实现偏离方案** | 功能存在但与方案设计不一致，可能导致行为偏差 |
| **P2** | **超出方案/方案模糊** | 代码超出方案范围（需评估），或方案描述不清导致实现缺乏依据 |
| **P3** | **轻微偏差** | 命名/位置等细节与方案不一致但不影响功能 |

### 边界案例

| 场景 | 分级 | 理由 |
|------|------|------|
| 方案定义了接口 A 但代码中完全没有 | P0 | 核心缺失 |
| 接口存在但参数顺序与方案不一致 | P1 | 可能导致调用方出错 |
| 方案要求 POST 但实现为 GET | P0 | HTTP 语义不同 |
| 代码添加了方案未提及的缓存层 | P2 | 超出方案但可能合理 |
| 代码添加了方案未提及的鉴权检查 | P3（或降为不报） | 安全补充通常合理 |
| 方案写"调用服务 A"，代码调用了服务 B 达到相同效果 | P1 | 需要解释偏差原因 |
| 方案流程图有 5 步，代码只实现了 3 步 | P0 | 流程不完整 |
| 方案要求测试覆盖但无测试用例 | P1 | 测试要求未满足 |

## coherence module（连贯性）严重度映射

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| **P0** | **引用链断裂 / 数值矛盾** | 引用的文件不存在导致运行时失败；数值矛盾导致读者按错误信息行动 |
| **P1** | **一致性/陈旧风险** | 描述性矛盾、注册条目不完整、路径错误、跨入口声明矛盾、陈旧确定性声明 |
| **P2** | **结构/维护负担** | 文件错位、孤儿文件/资源、缺失交叉引用、索引不全 |
| **P3** | **改善建议** | 建议合并/拆分、轻微命名不一致、轻微陈旧 |

## extensibility module 严重度映射

extensibility module 的 finding 不是「问题」而是「推荐」，P0-P3 含义为：

| 级别 | 含义 | 判定标准 |
|------|------|---------|
| **P0** | **核心缺失** | 不做会严重限制项目价值；用户核心工作流断裂；竞品已普遍具备 |
| **P1** | **重要缺口** | 显著影响体验或竞争力；有明确用户需求 |
| **P2** | **排期实施** | 有价值但非紧迫；改善特定场景 |
| **P3** | **锦上添花** | 按需考虑；成本可能高于收益 |

### 判定问句

> P0 vs P1：「不做这个功能，用户的核心任务是否无法完成？」
> - 无法完成 → **P0**
> - 可以完成但体验差 → **P1**
>
> P2 vs P3：「这个推荐是否解决了一个真实的用户痛点？」
> - 是，但不紧急 → **P2**
> - 不明显 → **P3**

---

## 统计规则

- **"有 finding" 的门槛**：所有级别（P0-P3）均算
- **修复优先级**：P0 → P1 → P2 → P3
- **阻塞发布判定**：默认 P0 全部修复才能发布；P1 建议发布前修复
- **每个 finding 必须且只能有一个级别**

---

## 分级争议解决

多轮视角对同一问题分级不一致时，统一使用以下规则（所有 module 和输入模式通用）：

1. **多数决**：取多数轮次给出的级别
2. **平局取高**：票数相同时取较高级别（保守原则）
3. **仅 1 轮确认**：列入 `DEFERRED`（标注「仅单轮确认，待人工判断」），不进入修复

> 示例：3 轮分别给 P1/P1/P2 → P1（多数决）；2 轮给 P1/2 轮给 P2 → P1（平局取高）。
